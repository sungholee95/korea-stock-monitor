## Features

- Real-time notification over Telegram

- Chat-driven UI: register, list, and remove alerts through bot commands

- Secure credential storage on system keyring; no plaintext `.env`

- Identical alerts evaluated once and fanned out to every subscribed user; no per-subscriber recalculation

- SQLite database to store historical data


## Getting Started

### Prerequisites

- [Astral uv](https://docs.astral.sh/uv/)
- Python 3.12+
- A [Kiwoom Securities OpenAPI](https://openapi.kiwoom.com) account with App Key and App Secret
- A Telegram bot authentication token generated by [@BotFather](https://core.telegram.org/bots/features#botfather)


### Installation

1. Clone the project

    ```bash
    git clone https://github.com/sungholee95/korea-stock-monitor.git
    ```

2. Go to project directory

    ```bash
    cd korea-stock-monitor
    ```

3. Install dependencies

    ```bash
    uv sync             # default installation
    uv sync --no-dev    # skip development dependencies
    ```

4. Initialise the application (set up configs at `~/.ksmonitor/configs`)
    
    ```bash
    uv run main.py init
    ```

### Configuration

As a safeguard against potential supply-chain attacks, `ksmonitor` avoids plaintext files (such as `.env`) for credentials, and instead uses the system keyring, via the [`keyring`](https://github.com/jaraco/keyring) library.

1. Register your Kiwoom API + Telegram bot credentials on the keyring.
    ```python
    import keyring

    keyring.set_password("kiwoom", "<your_Kiwoom_AppKey_name>", "<your_Kiwoom_AppKey>")
    keyring.set_password(
        "kiwoom", "<your_Kiwoom_AppSecret_name>", "<your_Kiwoom_AppSecret>"
    )
    keyring.set_password("telegram", "<your_Telegram_bot_token_name>", "<your_token>")
    ```

2. Fill in the fields of `kiwoom.yaml` and `core.yaml` with `<your_Kiwoom_AppKey_name>`, `<your_Kiwoom_AppSecret_name>`, and `<your_Telegram_bot_token_name>` as appropriate. Paper trading credentials are optional.


## Usage

1. Run
    ```bash
    uv run main.py
    ```
    to start the client and the Telegram bot.

2. From Telegram, send `/start` to your registered bot to get started.

### Available Telegram commands

| Command | Effect |
|---------|--------|
| `/start`, `/help` | Show available commands |
| `/alert <type> <args...>` | Register an alert; with no arguments, lists available alert types |
| `/unalert <type> <args...>` | Unregister an alert |
| `/status` | List the current chat's registered alerts |
| `/stop` | Unregister all alerts for the current chat |

For example, the following command sent to the bot will register an alert on stocks whose trade value exceeds 10 billion KRW within a 5-minute window, with a 30-minute per-ticker cooldown.
```
/alert TradeValue 10000000000 5 30
```
The full signature of `TradeValue` is 
```
TradeValue <threshold_krw> <window_minutes> <cooldown_minutes>
```
where `window_minutes` and `cooldown_minutes` are optional and default to `1` and `0`, respectively.


## Development

### How It Works

[main.py](main.py) is the orchestrator. It runs three coroutines concurrently:

1. **REST poll loop**: `KiwoomClient.poll_loop` polls every subscribed Kiwoom endpoint at `rest_poll_rate` and feeds it to alerts subscribed to that endpoint via `alert.ingest()`.
2. **Alert dispatch loop**: `dispatch_alerts` consumes `(alert, message)` tuples from `KiwoomClient.alert_loop` (an async generator) and forwards them to `TelegramBot.notify`, which fans out to every chat subscribed to that alert. Alert evaluation runs on each alert's own schedule (`is_due` / `next_eval_time`), decoupled from REST poll rate.
3. **Telegram bot**: `TelegramBot.start_bot` runs the long-poll updater to allow users to interact with the application via chat commands.

```
Kiwoom API ────────── KiwoomClient.poll()
                              │
              ┌───────────────┴────────────────┐
   KiwoomClient.poll_loop           KiwoomClient.alert_loop
              │                                │
         alert.ingest()                   alert.is_due()
                                               │
                                          alert.check() ───── yield (alert, message)
                                                                │
                                                         dispatch_alerts()
                                                                │
                                                       TelegramBot.notify(alert, msg)
```


### Extending `ksmonitor`

#### Adding a new API Endpoint

Every alert relies on an endpoint as its source of data. Each endpoint is composed of three cooperating classes bound to one `KiwoomEndpoint` enum member:

* `Request` — HTTP method, query parameters, header construction
* `Response` — return-code validation and pagination state
* `ResponseOutput` — typed parsing of the response body

When a `Request` subclass is defined, it self-registers via `__init_subclass__` of the base class, allowing straightforward lookup by the respective `KiwoomEndpoint` member.


#### Adding a new Alert

To define a new alert, subclass `BaseAlert`, provide the [endpoint](#adding-a-new-api-endpoint) that feeds the pertinent data and implement the following abstract methods:

- `ingest`: process the data polled by the client
- `evaluate`: determine if the alert should fire
- `format_message`: generate the message string to send over Telegram
- `spec`: positional constructor args, doubling as the value-based identity key (used to de-duplicate alert instances) and the serialisation payload for persisting user configs
